## Tool Choice

I chose **Claude Code** because it runs natively in the terminal alongside the project directory. There is no copy-paste step between an AI chat window and an editor — Claude Code reads, creates, and edits files directly. Every prompt and response is part of the same workflow, making it the most natural "vibe coding" environment: describe what you want, watch the file appear.

## Prompting Strategy

I started with a single high-level architecture prompt:

> "Build a Mini Vision Transformer in PyTorch. Patch size 16, image size 224, embed dim 768, 12 attention heads, 6 transformer layers, 10 output classes. Forward pass only — no training. Use a Conv2d patch projector, learnable class token, learnable positional encoding, and a Linear classification head. Print the output shape in main()."

Rather than asking for one class at a time, I gave Claude the full architectural spec in one shot. This mirrors how a senior engineer would hand off a task — with enough context to make good design decisions without micromanaging.

For the test file I prompted:

> "Write pytest shape-assertion tests for each module: PatchEmbedding, MultiHeadSelfAttention, TransformerBlock, and VisionTransformer."

## What Worked Well

Claude generated all five modules with correct tensor shapes on the first attempt. The PatchEmbedding `Conv2d` trick (kernel=patch_size, stride=patch_size as a linear projector) was used without prompting — Claude knew the standard ViT implementation pattern. Weight initialisation with `trunc_normal_` was also included unprompted.

## What Didn't Work

The implementation plan was structured task-by-task (one module per commit), which required multiple prompt rounds rather than a single generation. This was a deliberate TDD choice, but it meant ~5 rounds instead of the minimum possible 1-2.

## Iterations

- **Rounds 3–7:** One module per task (PatchEmbedding → MHSA → TransformerBlock → VisionTransformer → VIBE_LOG)
